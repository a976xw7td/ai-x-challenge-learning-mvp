## P3 范围（白皮书 §12）

P3 共 5 项，按依赖关系分三组：

### 第一组：基础设施（先做，无外部依赖）

#### T1：消息总线替换 Redis → Hermes/OpenClaw（ADR-001）

**目标：** 换传输层，不改 Envelope、不改 handle_message、不改业务逻辑。

**操作步骤：**

1. 调研 Hermes/OpenClaw 的入队/消费 API（预计是某种消息队列协议，类似 Redis Stream 的 XADD/XREADGROUP）
2. 在 `lib/server/` 下新建 `bus-adapter.ts`，抽象出统一的 `publish()` 和 `subscribe()` 接口
3. 实现 Redis 适配器（包装现有 `redis-stream.ts`）
4. 实现 Hermes 适配器（SDK 调用）
5. 过渡期双写：`publish()` 同时写 Redis + Hermes，consumer 两端都消费，对比结果
6. 稳定后切纯 Hermes，Redis 回归纯缓存/去重/限流角色
7. 验收：`/api/submit` 经 Hermes 总线返回 task_id，consumer 正常处理，FLUSHALL Redis 后提交仍正常（证明 Redis 不是必经之路）

**不改的文件：** `message-handler.ts`、`workflow.ts`、`review-workflow.ts`、`init-bus.ts`（handler 注册逻辑不动，只换 transport 层）

#### T2：Agent 注册中心

**目标：** `AGENT_TO_SP` 从写死常量 → 动态注册。Agent 启动时注册，关闭时注销。

**操作步骤：**

1. 飞书新建一张 `AgentRegistry` 表（字段：agent_id, api_key, status, capabilities, registered_at, last_seen_at）
2. 实现 `POST /api/agents/register`：Agent 启动时调用，携带 agent_id + api_key + capabilities
3. 实现 `DELETE /api/agents/:id`：Agent 关闭时注销
4. 实现 `GET /api/agents`：管理员查看在线 Agent
5. 修改 `agentToSP()`：先查飞书注册表 → 找不到回退到写死的 `AGENT_TO_SP`
6. WebApp 登录时自动注册学生 Agent（已有 API Key 生成逻辑，补注册步骤）
7. 验收：张浩登录 → Agent 自动注册 → `/api/agents` 能看到 `student-companion-zhanghao-001` online → 提交作业正常

**不改：** `checkTrust()` 签名、`RELATIONSHIPS` 结构（信任关系仍是配置，只改身份登记方式）

### 第二组：业务扩展（依赖 T1/T2）

#### T3：统一登录

**目标：** 学号+姓名登录 → 统一认证（对接 NSEAP 或自建）

**操作步骤（分阶段）：**

- **P3a（自建过渡）：** 支持多班级——Students 表加 `class_id` 字段，登录时自动识别班级；Session 支持跨班级
- **P3b（对接 NSEAP）：** 对接 NSEAP 统一认证 API，`getPrincipal()` 从 NSEAP token 解析（不再查飞书 Students 表做身份验证，飞书仅做业务数据）

#### T4：Hermes/WorkBuddy 桌面端适配

**目标：** 现有的 API Key + Envelope 方案对接真实桌面 App。

**操作步骤：**

1. 确认 Hermes 桌面端 / WorkBuddy 桌面端的通信协议（HTTP? MCP? gRPC?）
2. 根据协议写 Adapter（`lib/server/adapters/hermes-adapter.ts`、`workbuddy-adapter.ts`）
3. Adapter 职责：接收桌面端请求 → 构造 Envelope v2 → 入队（经 T1 总线）
4. 验收：WorkBuddy 桌面端发布 Challenge → 经总线 → 写飞书 → 群公告

#### T5：多班级多角色

**目标：** 支持多班级 + 助教/评委/观察员角色。

**操作步骤：**

1. Students/Challenges/Submissions 表加 `class_id` 字段
2. 查询 API 按 `class_id` 过滤（学生只看自己班，教师看自己教的班）
3. ServicePrincipal role 枚举扩展：`"ta" | "judge" | "observer"`
4. 飞书 Bot 通知按班级隔离（不同班发到不同群）
5. 验收：两个班各自发布 Challenge → 各自学生只能看到自己班的 → 互不干扰
